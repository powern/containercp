# MyVestaCP → ContainerCP: Site Import Plan

## 1. Call Graph: Створення сайту в ContainerCP

```
API POST /api/sites/create
    │
    SiteCreateOperation::execute(owner, domain, node, dry_run, profile, jobs, job_id)
    │
    ├── 1. Validation
    │   ├── Validator::validate_username(owner)
    │   └── Validator::validate_hostname(domain)
    │
    ├── 2. Check duplicate
    │   └── sites_.find(domain) — якщо є → помилка
    │
    ├── 3. Create site record
    │   └── sites_.create(domain, owner, node_id, web_server)
    │       └── SiteManager::create() → site.id = next_id_++
    │
    ├── 4. Create domain record
    │   └── domains_.create(domain, 0, site.id)
    │
    ├── 5. Create database record

    │   ├── db_user = safe + "_user"
    │   ├── PasswordGenerator::generate() → db_password
    │   └── databases_.create(db_name, db_user, db_password, 0, site.id)
    │
    ├── 6. HostingProvider::create_site(site, progress)
    │   │
    │   DockerComposeProvider::create_site(site, progress)
    │   │
    │   ├── 6.1. rt_.check_compose()
    │   ├── 6.2. SiteLayout::create()

    │   ├── 6.3. EnvGenerator::generate(domain, owner, db_name, db_user, db_password)
    │   │   Створює .env з: DB_NAME, DB_USER, DB_PASSWORD, MYSQL_ROOT_PASSWORD,
    │   │   REDIS_PASSWORD, SITE_DOMAIN, SITE_OWNER, TZ
    │   ├── 6.4. ComposeGenerator::generate(...)
    │   │   Створює docker-compose.yml з сервісами:
    │   │   ├── web (nginx:alpine або httpd:alpine)
    │   │   ├── php (php:8.x-fpm)
    │   │   ├── mariadb (mariadb:lts)
    │   │   └── redis (redis:alpine)
    │   ├── 6.5. Генерація конфігу веб-сервера (default.conf)

    ├── 7. Proxy setup

    │   └── proxy_provider_.reload()

    │
    └── 8. Rollback on failure
        ├── fs_.remove_directory(cfg_.sites_dir() + domain)
        ├── proxy_provider_.remove_proxy(domain)
        ├── docker compose down --volumes
        ├── docker network rm
        ├── databases_.remove(id)
        ├── domains_.remove(id)
```

## 3. Архітектура VestaSiteImporter

### 3.1 Новий клас: VestaSiteImporter

```cpp
namespace containercp::migration {

class VestaSiteImporter {
public:
    struct Options {
        std::string backup_path;       // Шлях до архіву
        std::string domain;            // Домен для імпорту
        std::string target_domain;     // Новий домен (опціонально)
        std::string owner;             // Власник сайту
        std::string node;              // Нода
        bool dry_run = false;          // Тільки показати що буде зроблено
        bool keep_staging = false;     // Не видаляти staging після імпорту
        bool skip_db = false;          // Пропустити імпорт БД
        bool overwrite = false;        // Перезаписати існуючий сайт
        std::string database;          // Примусово вказати БД (опціонально)
    };

    struct Manifest {
        // Інформація з архіву
        std::string domain;
        std::string web_archive_path;
        size_t web_size;
        bool has_wp_config = false;
        std::string wp_db_name;
        std::string wp_db_user;
        std::string wp_db_password;
        std::string wp_db_host;
        std::string found_db_name;
        std::string found_db_dump_path;
        size_t db_dump_size = 0;
        std::vector<std::string> all_databases;  // Всі БД домену
    };

    VestaSiteImporter(
        site::SiteManager& sites,
        domain::DomainManager& domains,
        database::DatabaseManager& databases,
        proxy::ReverseProxyManager& proxies,
        proxy::ProxyProvider& proxy_provider,
        provider::HostingProvider& provider,
        filesystem::Filesystem& fs,
        config::Config& cfg
    );

    Manifest inspect(const std::string& backup_path, const std::string& domain);
    core::OperationResult execute(const Options& opts);
    Manifest dry_run(const Options& opts);

private:
    // Helpers
    bool extract_web_files(const std::string& backup, const std::string& domain,
                           const std::string& dest);
    bool find_wp_config(const std::string& dest, std::string& out_db_name,
                        std::string& out_db_user, std::string& out_db_password,
                        std::string& out_db_host);
    bool find_db_in_archive(const std::string& backup, const std::string& db_name,
                            std::string& out_path, size_t& out_size);
    bool import_db_dump(const std::string& dump_path, const std::string& db_name,
                        const std::string& db_user, const std::string& db_password);
    bool update_wp_config(const std::string& dest, const std::string& db_name,
                          const std::string& db_user, const std::string& db_password);
    std::string make_staging_dir();
    void cleanup(const std::string& staging_dir);
    std::string normalize_db_name(const std::string& raw, const std::string& owner);

    // Members
    site::SiteManager& sites_;
    domain::DomainManager& domains_;
    database::DatabaseManager& databases_;
    proxy::ReverseProxyManager& proxies_;
    proxy::ProxyProvider& proxy_provider_;
    provider::HostingProvider& provider_;
    filesystem::Filesystem& fs_;
    config::Config& cfg_;
};
```

## 4. Повний алгоритм імпорту

### 4.1 inspect() — Dry-run / Manifest

```python
1. Перевірити backup_path — файл існує, .tar
2. Перевірити domain — валідний формат
3. tar -tf $backup | grep "^./web/$domain/domain_data.tar.gz"
   → Якщо не знайдено → помилка "Domain not found in backup"
4. Отримати розмір domain_data.tar.gz
5. tar -tf $backup | grep "^./web/$domain/" (список всіх файлів)
6. tar -tf $backup | grep "^./db/" → список БД в архіві
7. Створити Manifest { domain, web_archive_path, size, ... }
8. Якщо знайдено wp-config.php (п.4.2) → заповнити wp_* поля
9. Повернути Manifest
```

### 4.2 find_wp_config() — Пошук wp-config.php

```python
1. tar xf $backup -C $staging_dir ./web/$domain/domain_data.tar.gz
2. tar xzf $staging_dir/web/$domain/domain_data.tar.gz \
       -C $staging_dir/wp-scan/ --to-stdout "*/wp-config.php" 2>/dev/null
   АБО:
   tar -tf $staging_dir/web/$domain/domain_data.tar.gz | grep "wp-config.php"
3. Якщо знайдено:
   - Витягти wp-config.php в тимчасову директорію
   - Прочитати регуляркою (підтримуючи одинарні та подвійні лапки, пробіли):
     define\s*\(\s*['"]DB_NAME['"]\s*,\s*['"](.+?)['"]\s*\)
     define\s*\(\s*['"]DB_USER['"]\s*,\s*['"](.+?)['"]\s*\)
     define\s*\(\s*['"]DB_PASSWORD['"]\s*,\s*['"](.+?)['"]\s*\)
     define\s*\(\s*['"]DB_HOST['"]\s*,\s*['"](.+?)['"]\s*\)
   - НЕ виконувати PHP-файл (не використовувати php -r або eval)
   - НЕ логувати DB_PASSWORD (маскувати як ***)
   - Якщо DB_NAME визначається через змінну/getenv або не знайдено → вимагати --database
   - Нормалізувати DB_NAME: прибрати префікс старого користувача Vesta
   - Якщо DB_HOST != 'localhost' → попередити
4. Якщо не WordPress (немає wp-config.php):
   - Скасувати імпорт БД
   - АБО запитати користувача вказати БД вручну
```

### 4.3 find_db_in_archive() — Пошук дампу БД

```python
1. tar -tf $backup | grep "^./db/"
2. Список всіх баз: cut -f 3 -d / | sort -u
3. Для кожної БД:
   - Якщо назва БД містить wp_db_name або її нормалізовану версію → знайдено
   - Якщо знайдено → tar -tf $backup | grep "^./db/$found/" | grep "\.sql\.gz$"
4. Якщо не знайдено:
   - Показати список доступних БД
   - Користувач вказує --database вручну
```

### 4.4 execute() — Основний імпорт

```python
1. dry_run() → показати Manifest
2. Якщо sites_.find(domain) → error "Site already exists. Use --overwrite not supported in v1."
3. Створити staging: make_staging_dir()
4. Витягти та проаналізувати web files:
   tar xf $backup -C $staging_dir ./web/$domain/domain_data.tar.gz
   cd $staging_dir/web/$domain/
   # Визначити web root всередині domain_data.tar.gz:
   #   tar -tf domain_data.tar.gz | grep -E "^\.\/?(public_html|public|htdocs|www|root)\/"
   # Якщо знайдено public_html → web_root = public_html
   # Якщо знайдено public     → web_root = public
   # Якщо знайдено htdocs     → web_root = htdocs
   # Якщо знайдено www        → web_root = www
   # Якщо нічого не знайдено  → web_root = . (корінь архіву)
   tar xzf domain_data.tar.gz -C $staging_dir/web-root/ $web_root/
   # Результат: $staging_dir/web-root/ містить вміст web_root
   # Копіювати ВМІСТ web_root, а не саму директорію:
   #   cp -r $staging_dir/web-root/* /srv/containercp/sites/$domain/public/
   # Не допустити: public/public_html/
5. Знайти wp-config.php (п.4.2)
6. Якщо WordPress:
   - Знайти БД (п.4.3)
   - Витягти SQL dump:
     tar xf $backup -C $staging_dir ./db/$found_db/
     # .sql.gz знаходиться безпосередньо в db/<db>/ — без вкладеного db/
   - Запам'ятати старий DB_NAME, DB_USER, DB_PASSWORD
7. Створити сайт через SiteCreateOperation (dry_run=false):
   - SiteCreateOperation сам створює DatabaseRecord, генерує db_name, db_user, db_password
   - Після виконання знайти DatabaseRecord по site.id:
     for (const auto& d : databases_.list())
         if (d.site_id == site.id) { db_rec = &d; break; }
   - Використовувати db_rec->db_name, db_rec->db_user, db_rec->db_password
   - НЕ виконувати CREATE DATABASE / CREATE USER / GRANT — це зробив provider
8. Зупинити web + php (MariaDB і Redis залишити запущеними):
   docker compose -f /srv/containercp/sites/$domain/docker-compose.yml stop web php
9. Очистити public/:
   rm -rf /srv/containercp/sites/$domain/public/*
10. Скопіювати файли:
    rsync -a $staging_dir/web-root/ /srv/containercp/sites/$domain/public/
    # Перевірити: немає ../, немає symlink escape
11. Якщо WordPress:
    - Дочекатись готовності MariaDB:
      for i in 1..30; do
        docker exec site-{$site.id}-db mariadb -e "SELECT 1" && break
        sleep 1
      done
    - Перевірити SQL dump (gzip -t, наявність, тип mysql/mariadb):
      gzip -t $dump_path || error "Corrupt SQL dump"
    - Імпортувати дамп:
      gunzip -c $dump_path | docker exec -i site-{$site.id}-db mariadb $db_rec->db_name
      # Логувати stderr без паролів
    - Оновити wp-config.php: замінити старі DB_NAME, DB_USER, DB_PASSWORD на нові
12. Запустити stack повністю:
    docker compose -f /srv/containercp/sites/$domain/docker-compose.yml up -d
13. Health-check (без зовнішнього DNS, через containercp-proxy):
    curl -s -o /dev/null -w "%{http_code}" --resolve $domain:80:127.0.0.1 http://$domain/ 2>/dev/null
    # Враховувати 301/302 як success
14. Якщо keep_staging=false → rm -rf $staging_dir
15. Вивести результат
```

## 8. Error Handling та Cleanup

| Стадія | Помилка | Cleanup |
|--------|---------|---------|
| **inspect** | Архів не знайдено | — |
| **inspect** | Домен не знайдено в архіві | — |
| **inspect** | Пошкоджений tar | — |
| **inspect** | Сайт вже існує | Завершити без змін |
| **create site** | SiteCreateOperation failed | Вбудований rollback (видаляє все) |
| **extract files** | tar error | Rollback importer resources |
| **SQL dump** | gzip -t failed | Rollback importer resources |
| **SQL import** | MariaDB timeout | Rollback importer resources |
| **SQL import** | SQL error | Rollback importer resources |
| **wp-config** | Update failed | Import done, попередити |
| **start stack** | docker compose up failed | Rollback importer resources |
| **health check** | curl timeout/fail | Попередити, але сайт створено |
| **disk space** | < 2x необхідного | Не починати import |

### Алгоритм rollback (тільки ресурси створені importer-ом):
```cpp
void VestaSiteImporter::rollback(uint64_t site_id, const std::string& domain) {
    // 1. Docker compose down — тільки якщо stack був створений importer-ом
    if (stack_created_by_importer_) {
        std::string cmd = "cd " + cfg_.sites_dir() + domain
            + " && docker compose down --volumes --remove-orphans 2>/dev/null || true";
        std::system(cmd.c_str());
        // 2. Видалити мережу
        std::string net = "docker network ls --filter name=containercp-site-"
            + std::to_string(site_id) + " -q | xargs -r docker network rm 2>/dev/null || true";
        std::system(net.c_str());
    }
    // 3. Видалити директорію сайту (тільки якщо створено importer-ом)
    if (site_created_by_importer_) {
        fs_.remove_directory(cfg_.sites_dir() + domain);
    }
    // 4. Видалити proxy конфіг (тільки якщо створено importer-ом)
    if (proxy_created_by_importer_) {
        proxy_provider_.remove_proxy(domain);
    }
    // 5. Видалити records (тільки importer-ом створені)
    if (database_created_by_importer_) {
        for (const auto& d : databases_.list())
            if (d.site_id == site_id) databases_.remove(d.id);
    }
    if (domain_created_by_importer_) {
        for (const auto& d : domains_.list())
            if (d.site_id == site_id) domains_.remove(d.id);
    }
    if (site_created_by_importer_) {
        sites_.remove(site_id);
    }
    // 6. Видалити staging
    if (!keep_staging_) cleanup(staging_dir_);
}
```

### SQL dump перевірка:
```python
1. gzip -t $dump_path || error "Corrupt or invalid gzip: $dump_path"
2. Перевірити що всередині є SQL (перші байти після gunzip):
   gunzip -c $dump_path | head -1 | grep -qi "mysql\|mariadb\|CREATE\|INSERT" || warning
3. Визначити тип:
   case $TYPE in
       mysql|mariadb) import_via_mariadb ;;
       pgsql) warning "PostgreSQL not supported in v1, skip DB" ;;
       *) warning "Unknown DB type $TYPE, skip DB" ;;
   esac
4. Логувати stderr БЕЗ паролів (маскувати DB_PASSWORD перед записом)
5. Перевірити exit code pipeline:
   gunzip -c $dump_path | docker exec -i site-{$id}-db mariadb $db_name
   if [ ${PIPESTATUS[0]} -ne 0 ] || [ ${PIPESTATUS[1]} -ne 0 ]; then error; fi
```

### MariaDB readiness:
```python
for i in 1 2 3 ... 30; do
    docker exec site-{$id}-db mariadb -e "SELECT 1" >/dev/null 2>&1 && break
    sleep 1
done
if [ $i -eq 30 ]; then error "MariaDB not ready after 30s"; fi
```

## 9. Security Requirements

1. **Path traversal**: При розпаковці tar перевіряти що всі шляхи починаються з `./web/$domain/` або `./db/$db_name/`. Відхиляти `../` або абсолютні шляхи.
2. **Symlink escape**: При копіюванні файлів використовувати `rsync -a --safe-links` або перевіряти symlink targets.
3. **Max size**: Ліміт на розмір файлів (наприклад, 2 GB на сайт).
4. **Disk space**: Перевіряти `df` перед початком.
5. **Shell injection**: Всі зовнішні команди (`tar`, `gzip`, `docker exec`, `rsync`) через `CommandExecutor`, не через `std::system` з конкатенацією.
6. **DB password**: Маскувати в логах. Не використовувати старий пароль Vesta.
7. **Backup path**: Тільки дозволені шляхи (`/backup/`, `/srv/containercp/backups/`).
8. **Dry-run**: Жодних змін у системі.

## 13. Важливі зауваження

### 13.1 Бази даних: префікс користувача Vesta
VestaCP створює БД з префіксом користувача: `admin_wp_db` → при імпорті треба:
1. Видалити префікс `admin_` → `wp_db`
2. Додати префікс ContainerCP: `wp_db` → `myblog_com_db` (через `StringUtils::sanitize`)

### 13.2 serialized data в WordPress
В MVP (той самий домен) serialized data не потребує змін.
При додаванні `--target-domain` у майбутньому — використовувати
`wp search-replace` через WP-CLI в PHP контейнері, не `sed`.

### 13.3 SSL
- При створенні сайту SSL не випускається
- Після перемикання DNS → `POST /api/ssl/issue` через Web UI
- Старі SSL сертифікати Vesta НЕ імпортуються (одноразова міграція)

### 13.4 Час виконання
Для середнього WordPress сайту (50 MB файли + 10 MB БД):
- Інспекція: ~2 сек
- Імпорт: ~30 сек (залежить від Docker pull)
- Health-check: ~5 сек

### 13.5 Що НЕ імпортується
- DNS зони (DNS-модуль ContainerCP ще не реалізовано)
- Поштові скриньки (Vesta → Exim, ContainerCP → Docker Postfix)
- Cron jobs (немає аналога)
- FTP користувачі (не підтримується)
- Старі SSL сертифікати
- Статистика / логи відвідувань
