Summary
-------
The migration is not failing in Stage 2. Stage 2 has not started at all.

Current backend state says Stage 2 is available: can_import_files=true.
The GUI message "Run Analyze backup again to continue with Stage 2" is produced by the frontend after Stage 1, and the current implementation requires the operator to run Analyze again so the frontend can re-read the migration marker and render the Import files button.

This is a UX/state-machine orchestration problem, not a Site ID mismatch, not stale marker corruption, and not a hidden backend import failure.

4. Intentional Or Defective?
----------------------------
Current behavior is partially intentional but defective from UX/orchestration standpoint.

The backend is intentionally staged:

Analyze -> Create Site -> Analyze again -> Import files -> Import SQL

But the frontend after Stage 1 does not automatically re-run Analyze and does not render the Stage 2 button. It only prints:

Run Analyze backup again to continue with Stage 2.

That is confusing because the operator sees Stage 1 completed but no direct Stage 2 action.

16. Should Clicking Analyze Again Safely Continue?
--------------------------------------------------
Clicking Analyze again should be safe.

It calls:

POST /api/migration/vesta/inspect

That endpoint is designed as read-only/dry-run:

opts.dry_run = true;
auto manifest = importer.inspect(opts);

It should not duplicate Site or database creation. It should re-read the marker and return:

"can_import_files": true

Then the GUI should render the Import files button.


17. Would Clicking Analyze Again Duplicate Site/Database Creation?
------------------------------------------------------------------
No.

Analyze does not create Site/database/Docker resources.

The dangerous duplicate path would be clicking the old Stage 0 Create site / Import site button again. Even then, backend create-site re-runs inspect and rejects if manifest.site_exists is true:

if (manifest.site_exists) {
    r.status_code = 409;
    r.body = "{\"success\":false,\"error\":\"Site already exists. Import not possible.\"}";
    return r;
}

So duplicate creation should be blocked by backend.


18. Risk Of Repeated Analyze
----------------------------
Low.

Risks:

Re-runs tar listing on a 51GB archive.
Creates/removes temporary staging during wp-config/root detection.
Adds normal journal log entries.
Does not change marker, SQLite, Docker, document root, or database contents.

The biggest practical risk is operator confusion, not data mutation.


19. Recommended Minimal Fix
---------------------------
Minimal UX fix:

After successful create-site, the frontend should immediately call Analyze again or render the next action from a backend response that includes Stage 2 eligibility.

Best minimal change:

After importMigrationSite() succeeds, call analyzeBackup() automatically.

This would:

Refresh marker-derived state.
Replace the stale Stage 0 card.
Show Import files immediately when can_import_files=true.
Avoid telling the operator to manually run Analyze again.

Better backend/API improvement:

Make POST /api/migration/vesta/create-site return enough state to render next action:

{
  "migration_marker_found": true,
  "migration_stage": 1,
  "files_pending": true,
  "files_imported": false,
  "sql_pending": true,
  "can_import_files": true,
  "can_import_sql": false
}

Then frontend can render Import files directly without a second Analyze.

Important state persistence improvement:

The marker should persist the selected backup filename/path and analysis metadata, because currently .containercp-migration.json stores only:

domain, owner, site_id, stage, files_pending, files_imported, sql_pending

It does not store:

backup filename
web_archive_path
web_root_type
db_type
detected source DB

Because of that, a page reload loses enough UI state that the operator must manually reselect the backup/domain/owner before continuing.
