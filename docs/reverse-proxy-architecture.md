# Reverse Proxy Architecture

## Overview

ContainerCP manages a central nginx reverse proxy that routes external
HTTP/HTTPS traffic to the correct internal services (sites' web
containers, admin panel).  The entire system is self-contained —
nginx configuration is generated by ContainerCP core, never edited
manually.

```
Browser → https://example.com/
  → containercp-proxy (nginx:alpine, ports 80/443)
    → site-N-web:80 (per-site Docker Compose service)
```

```
Browser → https://admin.domain/
  → containercp-proxy (nginx:alpine, ports 80/443)
    → Web UI (172.17.0.1:8081)
```

## Components

### NginxProxyProvider (`libs/proxy/NginxProxyProvider.h/.cpp`)

The core class that manages the nginx proxy lifecycle:

- **Central proxy container**: creates, validates, and maintains the
  `containercp-proxy` Docker container (nginx:alpine).
- **Per-domain configs**: writes nginx server blocks to
  `<data_root>/proxy/sites/<domain>.conf`, each mounted into the
  container at `/etc/nginx/conf.d/`.
- **Configuration validation**: runs `nginx -t` inside the container
  via `test_config()`.  Results are cached in `cached_test_` to avoid
  running `nginx -t` on every health check.
- **Reload**: `docker exec containercp-proxy nginx -s reload`.
- **SSL integration**: `attach_certificate()` and `detach_certificate()`
  regenerate configs with HTTPS server blocks and validate before
  reloading (transactional rollback on failure).

Key methods:

| Method | Purpose |
|--------|---------|
| `ensure_central_proxy()` | Create or repair the proxy container |
| `create_proxy()` | Write a new domain config |
| `remove_proxy()` | Delete a domain config file |
| `test_config()` | Run `nginx -t`, cache result |
| `reload()` | `nginx -s reload` with readiness wait |
| `attach_certificate()` | Add HTTPS + optional redirect config |
| `detach_certificate()` | Remove HTTPS config, back to HTTP-only |
| `central_proxy_running()` | Check if container is in Running state |

### ProxyViewService (`libs/proxy/ProxyViewService.h/.cpp`)

Read-only view service that enriches proxy data for the API:

- Combines data from `ReverseProxyManager`, `SiteManager`,
  `CertificateStore`, and `SiteRuntimeManager`.
- `build_enriched_json()` — returns per-entry data with site name,
  SSL status, runtime backend health, protection flag, and
  allowed actions.
- `build_health_json()` — returns global health including container
  state, cached config test result, recovery status, and entry counts.

### ReverseProxyManager (`libs/proxy/ReverseProxyManager.h/.cpp`)

In-memory CRUD manager for `ReverseProxy` records:

| Field | Type | Description |
|-------|------|-------------|
| `domain` | string | The proxied domain name |
| `site_id` | uint64 | FK to Site (0 = admin panel) |
| `provider` | string | Always "nginx" |
| `config_path` | string | Path to `.conf` file on disk |
| `upstream` | string | Target e.g. `site-3-web:80` |
| `enabled` | bool | Whether the proxy is active |
| `status` | string | State text ("active", "disabled", "error") |

Records are persisted in `<data_root>/database/reverse_proxies.db`
(pipe-delimited text).

### ProxyConfigBuilder (`libs/proxy/ProxyConfigBuilder.h/.cpp`)

Generates nginx server block text:

- `build(params)` — dispatches to the correct block type
- `build_http_block()` — `server { listen 80; location / { proxy_pass ... } }`
- `build_https_block()` — same with `listen 443 ssl` + cert paths
- `build_redirect_block()` — `server { listen 80; return 301 https://...; }`

## Integration with ServiceRegistry

```
ServiceRegistry
  ├── proxy_provider_        (NginxProxyProvider)
  ├── reverse_proxies_       (ReverseProxyManager)
  ├── proxy_view_            (ProxyViewService)
  │     ├── reads: ReverseProxyManager, SiteManager,
  │     │         CertificateStore, SiteRuntimeManager
  │     └── used by: ApiServer (/api/proxy endpoints)
  ├── recovery_manager_      (RecoveryManager)
  │     └── calls: ensure_central_proxy(), ensure_admin_proxy(),
  │                 sync_all_https_configs()
  └── site_runtime_          (SiteRuntimeManager)
        └── used by ProxyViewService for backend status
```

## RecoveryManager interaction

`RecoveryManager` monitors proxy health and triggers recovery:

1. Every 60s: checks `central_proxy_running()`
2. If unhealthy: calls `ensure_central_proxy()` (recreates container)
3. Calls `ensure_admin_proxy()` (regenerates admin config)
4. Calls `sync_all_https_configs()` (resyncs site HTTPS configs)

`RecoveryManager` is intentionally separated from `Bootstrap`:
- Bootstrap handles initial setup (hostname, setup wizard).
- Recovery handles runtime proxy failures.
- They never overlap (Bootstrap is a separate process).

## SiteRuntimeManager integration

`ProxyViewService` uses `SiteRuntimeManager::get_status()` to
populate the Backend column.  For each proxy entry with `site_id > 0`,
the web service status (Running, Stopped, Unhealthy, etc.) is fetched
from the same `docker compose ps` + `docker inspect` pipeline used
by the Sites page.  No duplicate runtime logic.

For admin entries (`site_id == 0`), "Admin UI" is displayed (no
runtime check needed).

## Admin proxy

The admin panel proxy entry (site_id=0) is:
- Created automatically by `ServiceRegistry::start()`.
- Always regenerated on daemon startup.
- Protected: cannot be removed via the API (returns HTTP 403).
- Upstream: Docker gateway IP (172.17.0.x) port 8081 (Web UI server).
- Shows "System" badge and "Admin UI" in the Backend column.

## Site proxy entries

Site proxy entries are:
- Created during site creation by `DaemonApp`.
- Upstream: `site-{id}-web:80` (the site's web container).
- Not protected: can be removed via `POST /api/proxy/remove`.
- Backend status shows the site's web container runtime state.

## Configuration status

After daemon startup, the config test status is "Not tested" (cached
default).  The first `POST /api/proxy/test` call runs `nginx -t` and
updates the cache.  This avoids running Docker commands on every
health check.

## API endpoints

### GET /api/proxy
Enriched proxy entry list with site name, SSL status, backend health,
protection flag, allowed actions.

### GET /api/proxy/health
Global health: container state, cached config test, recovery state,
entry counts.  Lightweight — no Docker commands.

### POST /api/proxy/test
Run `nginx -t` inside the container via `CommandExecutor`.  Updates
cached result.  Returns success/failure with diagnostic message.

### POST /api/proxy/reload
Validates config first (`test_config()`), then reloads nginx.
Returns 500 if config test fails.  Serialized with other proxy
operations via `proxy_op_mutex`.

### POST /api/proxy/sync
Regenerates all HTTPS proxy configs from current CertificateStore
state.  Calls `ServiceRegistry::sync_all_https_configs()`.

### POST /api/proxy/recover
Full proxy self-healing: `ensure_central_proxy()` →
`ensure_admin_proxy()` → `sync_all_https_configs()`.  Calls
`RecoveryManager::recover_now()` — returns "already in progress"
if another recovery is active.

### Operation serialization

A static `proxy_op_mutex` in `ApiServer.cpp` serializes Test,
Reload, Sync, and Recover.  `try_lock()` is used — if a conflicting
operation is running, HTTP 409 is returned immediately.

## Workflows

### Config generation

```
Admin/proxy setup
  → ProxyConfigBuilder::build(params)
    → build_http_block / build_https_block / build_redirect_block
  → Write to <data_root>/proxy/sites/<domain>.conf
  → Reload (nginx -s reload)
```

### Reload workflow

```
POST /api/proxy/reload
  → proxy_op_mutex.try_lock() (409 if locked)
  → test_config() (500 if config invalid)
  → reload() → docker exec nginx -s reload
  → return result
```

### Sync workflow

```
POST /api/proxy/sync
  → proxy_op_mutex.try_lock() (409 if locked)
  → sync_all_https_configs()
    → for each active cert: attach_certificate(domain, cert, key)
  → return success
```

### Recovery workflow

```
POST /api/proxy/recover
  → proxy_op_mutex.try_lock() (409 if locked)
  → RecoveryManager::recover_now()
    → ensure_central_proxy()     # recreate container if needed
    → ensure_admin_proxy()       # regenerate admin config
    → sync_all_https_configs()   # resync site HTTPS configs
  → return result

Background loop (every 60s):
  → central_proxy_running()?
    ├── Yes → reset fail count
    └── No  → recover() (same three steps)
```
