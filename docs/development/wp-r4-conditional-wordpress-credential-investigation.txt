Final classification
--------------------

C. Parser false positive

The live WordPress database credentials are top-level direct constants. They are
not actually inside a PHP conditional block.

The current detector marks DB_NAME, DB_USER, and DB_PASSWORD as conditional
because its heuristic scans the previous 180 bytes before each define() call and
matches the word "for" inside a PHP comment:

  /** The name of the database for WordPress */

That comment is not PHP control flow.

Parser decision path
--------------------

Relevant implementation:

  libs/wordpress/WordPressConfigDetector.cpp

The detector finds define(...) calls through find_define_calls(content).

Each found call stores:

  struct DefineCall {
      std::size_t offset = 0;
      std::string body;
      bool conditional = false;
  };

The conditional flag is set by:

  looks_conditional(content, define_offset)

Current heuristic:

  bool looks_conditional(const std::string& content, std::size_t define_offset) {
      const std::size_t start = define_offset > 180 ? define_offset - 180 : 0;
      std::string prefix = content.substr(start, define_offset - start);
      const std::array<std::string_view, 7> controls = {"if", "elseif", "else", "switch", "foreach", "for", "while"};
      for (std::size_t i = 0; i < prefix.size(); ++i) {
          for (auto control : controls) {
              if (has_identifier_at(prefix, i, control)) {
                  return true;
              }
          }
      }
      return false;
  }

Problems with this heuristic:

  - It scans raw text, not PHP structure.
  - It does not skip comments.
  - It does not skip strings.
  - It does not track braces.
  - It does not require define() to be inside an open control-flow block.
  - It treats any nearby standalone "for" as conditional, even inside comments.

The triggering token is line 30, column 30:

Once parse_field() sees call.conditional, it sets:

  result.conditional = result.conditional || call.conditional;

Then inspect_content() adds conditional_constant issues.

Any conditional field then causes the whole inspection to become ambiguous:

  inspection.mutability = WordPressCredentialMutability::Ambiguous;
  inspection.status = WordPressCredentialStatus::Ambiguous;

Parser review answers
---------------------

1. Which parser state marks a definition as conditional?
   DefineCall.conditional, set by looks_conditional(content, define_offset).

2. Are all definitions inside any conditional automatically ambiguous?
   Yes. Any field with conditional=true makes the whole inspection ambiguous.

3. Does the parser distinguish safe guard patterns from dynamic config?
   No. It does not model safe guards such as if (!defined('DB_PASSWORD')).

4. Does the detector record exact source range?
   No. WordPressConfigDetector records offset, body, and conditional, but no
   replaceable source range in the public inspection result.

5. Could ConfigUpdater safely replace this literal?
   For the live structure, yes. WordPressConfigUpdater has its own block-aware
   inside_conditional_block() and exact ReplacementSpan. It would not consider
   this top-level line conditional. Rotation never reaches the updater because
   the detector blocks earlier.

6. What caused the ambiguity?
   Parser false positive. Comment text is interpreted as a control-flow token
   by the 180-byte heuristic.

Code changes required
---------------------

Repository code should change, but production data should not.

Required fix:

  - Replace WordPressConfigDetector::looks_conditional() with structural
    conditional detection.
  - Ignore comments and strings.
  - Mark define() conditional only when it is inside an active control-flow
    block.
  - Prior closed blocks must not affect later definitions.
  - Comment words such as "for WordPress" must not affect conditional
    classification.

Proposed narrow design
----------------------

Do not broadly allow conditional definitions.

Use a block-aware scanner equivalent to the updater's current logic, preferably
shared to avoid detector/updater divergence:

  - Track single-quoted strings.
  - Track double-quoted strings.
  - Track line comments.
  - Track block comments.
  - Track PHP control tokens only outside strings/comments.
  - Maintain a brace stack.
  - Push a conditional frame only when { opens after a control keyword.
  - Pop frames on }.
  - Mark define() conditional only when at least one active stack frame is
    conditional.

Fail closed for:

  - if/else credential selection;
  - elseif;
  - switch;
  - foreach, for, while;
  - function body definitions;
  - try/catch;
  - dynamic expressions;
  - environment variables;
  - included credential files;
  - duplicate definitions;
  - concatenation;
  - function-call values.

For this phase, because the live config is top-level direct constants, the
safest fix is not to introduce guarded_direct_constant yet. Support for guarded
constants should be a separate explicitly scoped enhancement only if a live or
test case requires it.

Security risks
--------------

Main risk of a detector fix:

  Accidentally allowing truly conditional credential definitions.

Mitigation:

  - Keep fail-closed behavior.
  - Add tests proving actual conditional branches remain ambiguous.
  - Do not classify guarded constants as mutable in this fix.
  - Do not treat comments/strings as control flow.
  - Do not change updater semantics unless sharing the same structural scanner.
  - Do not weaken duplicate/dynamic/include/expression rejection.
