# ContainerCP v0.7.0

## Overview

ContainerCP v0.7.0 is the SQLite storage release. It completes the manual migration path from legacy TXT storage to SQLite, adds strict activation validation, and makes SQLite the active runtime backend after explicit operator activation.

This release does not start Phase 12 and does not introduce new product features outside the v0.7.0 storage release scope.

## Major changes

- SQLite backend is available for the supported runtime resource set after explicit activation.
- Legacy TXT storage is preserved as migration source and immutable archive data.
- `containercp storage migrate-to-sqlite` performs the manual TXT-to-SQLite migration.
- Daemon startup validates existing SQLite activation state but does not run migrations.
- SQLite startup failures fail closed and do not silently fall back to TXT.
- SQLite write failures are propagated to callers instead of being ignored.

## SQLite migration

Migration is explicit and operator-controlled:

```bash
containercp storage migrate-to-sqlite \
  --source /srv/containercp/database \
  --database /srv/containercp/database/containercp.db \
  --archive-root /srv/containercp/archive \
  --source-version v0.6.0 \
  --target-version v0.7.0 \
  --confirm
```

The migration workflow validates inputs, creates a staged SQLite database, applies schema migrations, imports legacy TXT data, verifies imported data before activation, creates and verifies an immutable legacy archive, publishes `containercp.db`, and writes `storage-state.json`.

## Activation behavior

SQLite activation requires both:

- A valid `/srv/containercp/database/storage-state.json` created by the migration command.
- Backend selection set to SQLite through the storage backend selector.

Activation is not automatic. Operators must not enable SQLite until migration succeeds.

## Startup validation

When SQLite is selected, daemon startup validates:

- activation-state schema and required fields;
- active backend value;
- database path consistency;
- migration ID and version fields;
- SQLite database presence and schema version;
- SQLite integrity and foreign-key checks;
- activation archive presence and manifest consistency;
- filesystem type, symlink, permission, and ownership safety.

Startup validation does not run schema or data migrations.

## Failure behavior

Invalid SQLite state causes daemon startup to fail closed. The daemon does not start API/UI listeners for invalid SQLite activation state and does not silently fall back to TXT storage.

SQLite write failures are surfaced to callers. Failed child-table writes are tested for rollback behavior so a rejected write does not silently corrupt the last committed SQLite state.

## Security validation

The v0.7.0 SQLite activation path rejects unsafe filesystem inputs before startup completes:

- symlinked database, activation-state, or archive paths;
- non-regular activation-state files;
- unsafe write permissions on storage directories, database files, activation-state files, and archive paths;
- ownership mismatches where startup safety checks can validate them;
- archive manifest/checksum inconsistencies.

## Compatibility notes

- SQLite activation is explicit.
- Existing TXT data is not silently deleted by the migration command.
- Daemon startup does not run migrations.
- Unsupported or invalid SQLite state causes startup failure.
- There is no automatic fallback to TXT after SQLite is selected.
- Legacy TXT files remain relevant as migration source and archive data.
- `site_id=0` remains valid for approved sentinel/system records.
- Existing v0.6.0 migration fixtures and historical release notes remain unchanged.

## Upgrade notes

For existing v0.6.0 installations, follow `docs/sqlite-activation-runbook.md`:

1. Confirm the legacy TXT installation is healthy.
2. Run `containercp storage migrate-to-sqlite` with `--source-version v0.6.0` and `--target-version v0.7.0`.
3. Review migration diagnostics and archive path.
4. Enable the SQLite backend.
5. Restart `containercpd`.
6. Confirm `STORAGE` startup validation logs report SQLite readiness.
7. Keep the immutable archive according to operational retention policy.

## Known limitations

- SQLite activation remains explicit and operator-driven.
- Archive retention and deletion are not automated.
- Rollback is configuration-based only while appropriate legacy TXT data or archive recovery data is retained.
- The release does not introduce automatic online migration during daemon startup.
- The release does not modify the database schema beyond the already-validated v0.7.0 SQLite schema.
- c-ares deprecated API calls remain behind local compatibility wrappers pending a dedicated API migration.
