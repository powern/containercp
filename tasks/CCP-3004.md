# CCP-3004: Fix filesystem rollback gap

## Description

When `containercp site create` fails (e.g., Docker not installed),
the site directory and its contents are created by
`DockerComposeProvider` before the failure, but the rollback in
`SiteCreateOperation` only removes in-memory resources. The
filesystem artifacts remain on disk.

## Requirements

### Root cause

`DockerComposeProvider::create_site()` creates the filesystem layout
before calling `runtime.create_site_stack()`. If the runtime call
fails, the filesystem is already created and never cleaned up.

### Solution options

**Option A:** Move filesystem creation into `SiteCreateOperation`,
so it can be rolled back along with in-memory resources.

**Option B:** Expose a cleanup method on `HostingProvider` that
removes the filesystem for a given site. `SiteCreateOperation`
calls this during rollback.

**Option C:** Check Docker availability before any filesystem
creation inside `DockerComposeProvider`. If Docker is unavailable,
return early without creating files.

### Recommendation

Option C is the simplest and safest: add an early Docker check at
the top of `DockerComposeProvider::create_site()` before creating
any files or directories.

## Acceptance criteria

1. When Docker is unavailable, `site create` does not create any
   files or directories
2. After a failed `site create`, the site directory does not exist
3. When Docker is available, behavior is unchanged
4. All existing commands still work
