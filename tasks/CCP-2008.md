# CCP-2008: Recovery support

## Description

If `site create` fails after creating some resources (e.g., site record
created but Docker unavailable), the system is left in a partial state.
Resources must be rolled back automatically.

## Requirements

### Rollback strategy

Wrap resource creation in a try/catch or error-checking sequence.
If any step fails:

1. Revert all previously created resources
2. Print a clear error message
3. Exit with non-zero code

### Steps that can fail

| Step | Resources created | Rollback action |
|------|------------------|----------------|
| 1. Validate | none | nothing |
| 2. Site record | Site + Domain + Database (in-memory) | remove from managers |
| 3. Filesystem | directories + files | remove directory |
| 4. Docker compose | docker-compose.yml | remove file |
| 5. Docker up | running containers | docker compose down |

### Implementation

- Use RAII-style guards or manual cleanup in `SiteCreateOperation`
- Store pointers/references to created resources so they can be removed
- Call cleanup if `create_site_stack()` returns failure
- Print: "Error: <reason>. Created resources have been rolled back."

### Safety

- Never delete existing data that was not created by this operation
- If rollback itself fails, log the error but do not abort

## Acceptance criteria

1. If Docker is unavailable, site record and directory are rolled back
2. After failed create, `site list` shows no new site
3. After failed create, filesystem has no new site directory
4. All existing commands still work
